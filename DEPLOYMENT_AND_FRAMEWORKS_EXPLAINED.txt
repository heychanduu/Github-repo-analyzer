SECTION 1: WHY SPLIT FRONTEND AND BACKEND HOSTING?
────────────────────────────────────────────────────────────────────────────────

At first glance it may seem redundant to use two separate hosting platforms (e.g.,
Firebase + Cloud Run, or Vercel + Render), but that is a deliberate and professional
architectural decision. Here is exactly why:

The project is a FULL-STACK APPLICATION made up of two very different parts:

  1. FRONTEND  — React + Vite app (purely HTML, CSS, JavaScript files)
  2. BACKEND   — Spring Boot app (a Java server that runs continuously)

These two parts have completely different technical requirements when hosted:

  FRONTEND needs:
    - Fast global delivery (CDN)
    - Static file serving (no server-side computation)
    - Instant deployment on every Git push
    - A domain with HTTPS

  BACKEND needs:
    - A real server process running 24/7
    - Java runtime environment (JVM)
    - Docker container support
    - Ability to handle HTTP API requests
    - Memory and compute resources

NO SINGLE FREE-TIER PLATFORM DOES BOTH WELL:

  • Vercel is PURPOSE-BUILT for frontends. It has a global CDN (Content Delivery
    Network), zero-config React/Vite support, and is extremely fast for static
    assets. BUT it does NOT support running a Java/Spring Boot server process.

  • Render supports long-running server processes (Node, Python, Java, Docker),
    making it ideal for the Spring Boot backend. BUT it is NOT optimised for
    frontend static delivery the way Vercel is.

BEST-OF-BOTH-WORLDS APPROACH:
  → Deploy the FRONTEND on Firebase or Vercel (what they are purpose-built for)
  → Deploy the BACKEND on Google Cloud Run or Render (purpose-built for containers)

This is the industry-standard pattern used by most real-world startups and
professional projects. The frontend talks to the backend via an environment
variable (VITE_API_URL) pointing to the Render URL, keeping them decoupled.

SECTION 2: WHAT IS VERCEL?
────────────────────────────────────────────────────────────────────────────────

VERCEL is a cloud platform for deploying frontend applications and static sites.

Key facts:
  - Founded by the creators of Next.js (a popular React framework).
  - Specialises in deploying JavaScript-based frontends (React, Vue, Next.js,
    Svelte, Vite, plain HTML/CSS/JS, etc.).
  - Uses a GLOBAL EDGE NETWORK (CDN) — your app files are cached in servers
    all around the world, so users get fast load times regardless of location.
  - Connects directly to your GitHub repo; every git push triggers an automatic
    deployment (CI/CD out of the box).
  - Automatically provisions an HTTPS SSL certificate.
  - Each pull request gets its own "preview deployment" URL.
  - Free tier available with no credit card needed.

How it works (simplified):
  1. You push code to GitHub.
  2. Vercel detects the push, runs the build command (e.g. `npm run build`).
  3. The resulting static files (HTML, JS, CSS) are distributed to Vercel's CDN.
  4. Users worldwide access those files from the nearest edge server.

Why Vercel for THIS project:
  - Our frontend is built with React + Vite, which Vercel auto-detects and
    configures without any manual setup.
  - Zero configuration needed — Vercel reads the Vite config and knows exactly
    what build command to run and which folder to serve.
  - The free tier is generous and perfectly suited for a portfolio project.

SECTION 3: WHAT IS RENDER?
────────────────────────────────────────────────────────────────────────────────

RENDER is a cloud platform for deploying backend services, APIs, databases,
and full Docker-based applications.

Key facts:
  - Think of it as a simpler, modern alternative to Heroku (which ended its
    free tier in 2022).
  - Supports many runtimes: Node.js, Python, Ruby, Go, Rust, Java, and DOCKER.
  - Can run LONG-RUNNING PROCESSES — unlike Vercel, Render keeps your server
    alive and responsive to incoming HTTP requests 24/7.
  - Also connects to GitHub for automatic deploys on every push.
  - Provides managed databases (PostgreSQL, Redis) as add-ons.
  - Free tier spins the service DOWN after 15 minutes of inactivity (cold start
    on next request), which is a known limitation of the free plan.

How it works (simplified):
  1. You push code to GitHub (or connect a Docker image).
  2. Render pulls the code, builds the Docker image using your Dockerfile.
  3. The container is started and kept running on Render's server infrastructure.
  4. Render gives you a public HTTPS URL for your service.
  5. API requests from the frontend are routed to this container.

Why Render for THIS project:
  - Our backend is a SPRING BOOT application — a Java server that must be
    running continuously to respond to API requests.
  - We have a Dockerfile in the project root that defines how to build the
    Spring Boot app into a container image.
  - Render has excellent Docker support, making it straightforward to deploy.
  - The free tier is sufficient for a portfolio/demo project where traffic is
    not extremely high.
  - It replaced Heroku as the go-to free backend hosting platform.

SECTION 4: WHAT IS REACT? (FRONTEND FRAMEWORK)
────────────────────────────────────────────────────────────────────────────────

REACT is a JavaScript library for building user interfaces, created by Facebook
(Meta) and open-sourced in 2013.

Core concept — COMPONENTS:
  - Everything in React is a "component" — a self-contained piece of UI.
  - Components can be thought of as custom HTML elements that manage their own
    data (called "state") and appearance.
  - Example: a <GitHubUserCard /> component that receives a username and renders
    the avatar, name, repositories, and stats for that user.

Core concept — VIRTUAL DOM:
  - React maintains a lightweight virtual copy of the browser's DOM (Document
    Object Model).
  - When data changes, React calculates the minimum set of changes needed and
    applies only those to the real DOM — making updates very fast.

Core concept — JSX:
  - React uses JSX syntax, which looks like HTML inside JavaScript files.
  - Example:
      function UserCard({ name }) {
        return <div className="card"><h2>{name}</h2></div>;
      }

Core concept — STATE and PROPS:
  - STATE is data that belongs to a component and can change over time
    (e.g. the list of fetched repositories).
  - PROPS are inputs passed from a parent component to a child component
    (e.g. passing a username down to a chart component).

Why React for THIS project:
  - The UI is highly DYNAMIC — showing charts, repository cards, language
    breakdowns, and contribution stats all based on API responses.
  - React's component model makes it easy to build reusable UI pieces
    (e.g. one RepoCard component reused for every repository in the list).
  - React has a massive ecosystem (charting libraries, icon packs, UI kits)
    that were leveraged in this project.
  - It is the most widely-used frontend library, making it a strong portfolio
    choice to demonstrate.

SECTION 5: WHAT IS VITE? (FRONTEND BUILD TOOL)
────────────────────────────────────────────────────────────────────────────────

VITE (pronounced "veet", French for "fast") is a modern build tool and
development server for JavaScript applications, created by Evan You (the creator
of Vue.js).

The problem it solves:
  - Traditional bundlers like Webpack must bundle ALL your JavaScript before
    the dev server starts — this gets slower the larger your app becomes.
  - Vite uses native browser ES Modules during development, so it only
    processes the files the browser actually requests — leading to near-instant
    server startup and hot module replacement (HMR).

Key features:
  - INSTANT DEV SERVER: starts in milliseconds regardless of project size.
  - HOT MODULE REPLACEMENT (HMR): changes appear in the browser in under 50ms
    without a full page reload.
  - OPTIMISED PRODUCTION BUILDS: uses Rollup under the hood to create highly
    optimised bundles for deployment.
  - BUILT-IN PROXY SUPPORT: in this project, Vite's proxy was used during
    development to forward /api requests to the local Spring Boot server,
    avoiding CORS issues.
  - ENVIRONMENT VARIABLES: supports .env files with VITE_ prefix, which is how
    the VITE_API_URL (pointing to the Render backend) is configured.

Why Vite for THIS project:
  - Fast development experience when building and iterating on the UI.
  - Zero-config React support (just install the React plugin).
  - The proxy feature made local development seamless — the frontend could talk
    to the backend without manually changing API URLs.
  - Vercel natively supports Vite projects with zero configuration.

SECTION 6: WHAT IS SPRING BOOT? (BACKEND FRAMEWORK)
────────────────────────────────────────────────────────────────────────────────

SPRING BOOT is a Java-based framework for building production-ready web
applications and REST APIs quickly, with minimal configuration.

Background — Spring Framework:
  - Spring is a massive, enterprise-grade Java framework (around since 2003)
    that provides everything needed to build large-scale applications.
  - It handles dependency injection, database access, security, web requests,
    and much more.

What Spring Boot adds:
  - Spring alone requires a lot of XML/Java configuration.
  - Spring Boot introduces "convention over configuration" — it provides
    sensible defaults so you can get a server running with minimal setup.
  - It has an EMBEDDED SERVER (Tomcat by default), meaning you do not need
    to install a separate web server — the app is self-contained and runs as
    a single JAR file.

Key components used in THIS project:

  spring-boot-starter-web:
    - Provides REST API capabilities.
    - Lets you annotate Java classes with @RestController and @GetMapping to
      define API endpoints (e.g. GET /api/github/{username}).

  spring-boot-starter-webflux (WebClient):
    - A reactive, non-blocking HTTP client.
    - Used to call the EXTERNAL GitHub API from the backend without blocking
      threads — much more efficient than traditional blocking HTTP clients.
    - This is how the backend fetches user data, repositories, commit history,
      and contribution stats from api.github.com.

  Maven (pom.xml):
    - The dependency management and build tool for the Java project.
    - Equivalent to npm's package.json but for Java.

Why Spring Boot for THIS project:
  - The GitHub API enforces rate limits and requires careful handling of
    pagination, token authentication, and parallel requests. Spring Boot's
    WebClient handles all of this elegantly.
  - Java is strongly typed, making it easier to model complex GitHub API
    responses into clear, structured Java objects.
  - Spring Boot is the industry-standard for Java backends and is a highly
    sought-after skill in the job market, making it a valuable portfolio choice.
  - It produces a single self-contained JAR that Docker can run easily —
    important for the Render deployment.

SECTION 7: WHAT IS DOCKER? (CONTAINERISATION)
────────────────────────────────────────────────────────────────────────────────

DOCKER is a platform for packaging applications into "containers" — lightweight,
isolated environments that include everything needed to run the application
(code, runtime, libraries, configuration).

Key concepts:
  - DOCKERFILE: a script that defines how to build a container image.
    This project's Dockerfile tells Docker how to compile the Spring Boot app
    and package it into a runnable image.
  - DOCKER IMAGE: the built, immutable snapshot of the application.
  - DOCKER CONTAINER: a running instance of an image.
  - DOCKER COMPOSE: a tool (docker-compose.yml) to run multiple containers
    together — in this project, it runs both the backend and (optionally)
    the frontend locally with a single command.

Why Docker for THIS project:
  - Ensures the app runs IDENTICALLY on every machine and on Render's servers.
    "Works on my machine" is no longer an excuse.
  - Render's deployment is Docker-native, so the Dockerfile is used directly
    in production.
  - Makes it easy for anyone cloning the repo to run the entire stack locally
    with just `docker-compose up`.

SECTION 8: HOW EVERYTHING FITS TOGETHER
────────────────────────────────────────────────────────────────────────────────

Here is the complete flow of the application:

  USER'S BROWSER
      │
      │  visits Vercel URL
      ▼
  VERCEL (CDN)
      │  serves React + Vite built files (HTML, JS, CSS)
      │  these files run inside the user's browser
      │
      │  user types a GitHub username in the UI
      │  React makes an API call using axios
      │
      ▼
  RENDER (Docker Container — Spring Boot)
      │  receives GET /api/github/{username}
      │  uses WebClient to call api.github.com
      │
      ▼
  GITHUB API
      │  returns JSON data (users, repos, commits, languages, etc.)
      │
      ▼
  RENDER  — processes and returns the data to the browser
      │
      ▼
  BROWSER — React updates the UI with charts, stats, and repo cards


ENVIRONMENT VARIABLE BRIDGE:
  - On Vercel, VITE_API_URL is set to the Render backend URL.
  - The axiosConfig.js file reads this variable and points all API calls
    to the correct Render URL in production (and uses a local proxy in dev).

SECTION 9: ALTERNATIVE (CURRENT) DEPLOYMENT: FIREBASE & CLOUD RUN
────────────────────────────────────────────────────────────────────────────────

While Vercel and Render are great free-tier options, this project is currently
live using Google Cloud Platform (GCP) for enhanced performance, scalability,
and a fully integrated ecosystem.

WHAT IS FIREBASE HOSTING? (FRONTEND)
  - Google's dedicated web hosting platform.
  - Serves static assets instantly via their global edge CDN.
  - Deploys seamlessly using GitHub Actions (CI/CD).
  - Provides a cleaner default URL (e.g. your-project.web.app).

WHAT IS GOOGLE CLOUD RUN? (BACKEND)
  - A serverless container platform by Google.
  - Takes our backend Docker image and runs it on Google's heavy infrastructure.
  - "Scales to zero" — meaning you only pay for the exact milliseconds the CPU
    is processing an API request, making it extremely cost-effective.
  - Super fast cold-starts compared to Render's free tier.
  - Configured with Continuous Deployment (CD) directly from our GitHub repo using
    Google Cloud Build.
